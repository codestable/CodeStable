# cs-clarify

洞察用户的真实诉求，而不是帮他整理说出来的那个。

`context/` 里的现状、实际的代码和 issue 有助于理解背景。读一下 `attention.md`，用全文搜索确认用户提到的关键词不会和已有术语打架。

## 关键区分

**方案与问题。** 用户说的几乎都是方案——"我想加一个缓存层"、"我们需要拆成微服务"。方案是对问题的某一种回应，不是问题本身。在没有确认问题之前评估方案，就像在没有看过病人之前开药。

**表面诉求与真实诉求。** 人们会说想要一匹更快的马。表面诉求是用户意识到并且说出来的；真实诉求往往藏在更深的地方，连用户自己都没有完全意识到。一个需求聊着聊着方向大变，通常不是用户善变，而是真实诉求终于浮出了水面。洞察的目标就是加速这个浮出的过程。

**偏好与约束。** "我喜欢 PostgreSQL"是偏好，"数据必须支持事务"是约束。偏好可以商量，约束不能绕过。很多争论是因为把偏好当成了约束，或者把约束当成了偏好。

## 原则

**让用户看见他自己看不见的东西。** 这是你在场的全部理由。可以是一个他没注意到的约束，一个他以为不相关的已有能力，一个他下意识回避的取舍。如果聊完之后他的认知和聊之前一样，这次澄清就没有发生。

**取舍比罗列更有信息量。** "这三个方向各有什么好处"不如"如果只能选一个你选哪个"。优先级不是问出来的，是在放弃的痛感里显现的。

**说不清"不做什么"就等于还没想清楚。** 做什么可以慢慢加，但边界现在就要画。范围蔓延都是从"顺便加一下"开始的。

**方案有明显毛病就直说。** 洞察不是迎合。看到解错了题、过度工程、有更轻的路径，不说出来就是放弃了澄清最核心的价值。但评估下来确实合理的，就直接建议动手，不要为了凑流程硬发散。

## 什么时候算聊完了

澄清结束的标志不是"聊了足够久"，而是以下三个条件同时满足：

1. **问题能用一句话说清楚**——用户点头确认，不是你单方面复述。
2. **做什么和不做什么都有了轮廓**——不需要精确，但边界要在。
3. **最大的未知已经被标出来**——不要求解决，但要求看见。

没有同时满足就继续聊。达到之后进入出口判断。

## 深挖和动手验证

概念反复打架收不住时，可以进入更密集的追问，最多三到五轮。遇到"得写起来才知道"的就跳过，用户一敷衍就退回收敛。

如果争的是事实不是偏好，结果会改变方向，半小时内能出结论——直接动手验证比继续聊更省时。

## 出口

聊完判断大小，选出口：

- 够清楚了 → 直接写（`cs-code`），需要追踪就开 issue
- 小需求，收敛了 → 直接写或开 issue
- 大需求，可以拆了 → 移交 `cs-epic`，带上要点
- 大需求，还想再想 → 存 `.codestable/clarify/{slug}/clarify.md`

判错了随时升降级。

### 走任何出口之前，先让共识归位

澄清的产物不是聊天记录，是共识。共识不落地，下游写代码、开 issue 时接不到，又得重聊。退出前提示用户走 `cs-context` 补两样（`cs-clarify` 自己不写）：

- **新澄清出的领域术语 / 统一的叫法** → 进词汇表。下游 `cs-code` 用同名概念命名，代码才接得上 context。
- **拍板了不好回退、真有取舍的决策**（选库、模块划分、明确不做某事）→ 进取舍说明。

还在摇摆没定的术语别进 context（它没有 draft），留 `clarify.md` 草稿里，落地后再毕业。

## 不要做的事

- 跳过判断大小。
- 替用户决定规模。拿不准就问。
- 处理已明确的 bug（→ `cs-issue`）。
- 自己动手写代码或拆 epic。阶段之间的确认是必须的。
- 共识没归位就走出口——澄清出的术语 / 取舍不落地，下游接不到。
