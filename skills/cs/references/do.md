# Do：实现明确改变

按用户已经明确授权的目标写代码并验证；有 issue 时回写，没有时不补造事项。

## 背景

CodeStable 里真正危险的不是“AI 不会写代码”，而是写着写着目标漂了：顺手加功能、顺手重构、发现判断不对还硬往前冲。另一个相反的危险，是为一个几分钟能完成的明确改变强制建立事项，让管理成本超过变化本身。

Do 模式的任务是按明确目标和范围实现，用与风险相称的证据证明结果。复杂或持续管理的工作沿 issue 推进并回写；简单、明确、一次完成的工作可以直接执行。如果实现中发现目标或设计不成立，就停下来说明，不把漂移藏进代码里。

## 原则

**只做已授权的改变。** 有 issue 时守住其范围；直接改变时守住用户明确目标和必要影响面。遇到额外功能、相邻 bug 或无关重构，不塞进本次变更；只有用户希望持续追踪时才建议新 issue。

**先选管理强度。** 用户明确要求直接改时直接行动。用户只表达目标、尚未授权实现，且是否留痕会实质改变后续工作时，说明推荐后询问“直接做还是先建 issue”。简单、明确、低风险、一次完成通常直接做；范围取舍、多轮推进、跨会话、交接、显著风险或长期质量承诺通常值得 issue。

**沿现状模型实施。** 开工前要能解释相关触发怎样产生结果，并知道变化直接落在哪里、哪些相邻行为需要验证。不能解释时先回到轻量 Explore / Design；现状跨越多个未知边界时升级为完整 Explore，不边写边猜。

**小偏差回写，大偏差停下。** 如果设计和代码现状冲突，小偏差可以更新 issue 的实现设计或执行记录后继续；大偏差要停止实现，说明 issue 边界或设计为什么不成立，建议切换到 Design 或 Talk 模式。

**先走最小实现梯子。** 开始写代码前，按 `economy.md` 先判断这个改动是否仍然需要存在，能否删除或收窄，仓库里正确归属是否已有 helper、类型、模块或同类模式，标准库、平台原生能力、数据库约束、CSS/HTML 或已安装依赖是否能覆盖。能复用就复用，能不写就不写，最后才写新的最小代码；不把一行难懂代码当作独立目标。

**最小不是草率。** 不要为了少写而跳过根因、输入校验、安全、可访问性、防数据丢失的错误处理、用户明确要求或必要验证。小而错不叫轻量，只是把复杂度推迟爆炸。

**渐进式实现。** 复杂实现可以先完成一段可验证的垂直切片或一个明确子步骤，跑对应验证，回写进展和未知，再继续下一段。停在清楚的中间态，比交出一堆无法解释的改动更好。

**验证要能说明问题。** 验证不是最后跑一下命令。bug 先复现或定位预期/实际分叉；feature 优先通过用户可观察行为、接口结果或集成路径验证；refactor/chore 验证行为不变。非平凡逻辑至少留下一个可运行、会在行为损坏时失败的最小检查；优先复用现有测试入口，不为了一个窄变化新建测试框架或 fixture 体系。

**质量目标逐项兑现。** Issue 已选中的质量目标是实现与关闭约束；直接改变也必须守住相关 spec 约束、用户要求和风险扫描发现的必要质量边界。直接改变不生成形式化九项清单，但验证仍要覆盖真正会改变正确性或失败代价的风险。

**UI 实现服从视觉契约。** Issue 或上层 spec 用线框图、流程图和标注规定了界面关系时，按 `ui-spec.md` 实现已确认的目标结构和关键状态。不要把图中的占位尺寸当像素要求，也不要为了现有组件方便静默改变稳定约束；发现图与文字、当前与目标冲突时先回到 Design / Spec 消歧。

## 行动指南

### 先锁定目标与管理方式

先确认用户已经授权实现，并判断当前工作是否已有目标 issue：

- 用户给出 issue 路径或明确要求实现某个 issue：读取并确认当前版本，按受管理路径推进。
- 当前上下文已经在明确 issue 中：延续该 issue，不重复选择。
- 用户明确要求直接修改、修复或实现，目标简单清楚：按直接路径推进，不创建 issue。
- 用户只是在讨论目标，是否立即改动并不清楚：切换 Talk，或用一句带推荐的问题确认直接做还是持续管理。

不要仅因为仓库存在 `.cs/issues/` 就把一次直接请求强行绑定到旧 issue；也不要在复杂工作明显需要持续管理时悄悄跳过边界和记录。

开始前先复用当前上下文。下面是必须掌握的信息，不是每次全量重读清单；只有没读过、有修改迹象、需要精确引用/写回，或本次实现需要新增局部时才补读。

- 目标与范围：直接改变以用户最新明确请求和当前上下文为准；受管理改变必须确认目标 issue 当前版本，掌握目标、范围、质量目标、设计、验证和关闭回写。
- 相关 spec：独立 issue 读取 `.cs/spec/` 的相关入口；epic issue 先读对应 epic 的 `spec.md`，必要时再读其引用的 project spec。
- 相关代码：确认实现设计仍然贴合代码现状。

先确认当前上下文或 issue 的 `## 现状如何工作` 足以解释本次相关路径；存在具体变化时，分清必须修改、需要验证和仍待调查。简单直接改变可以只在工作上下文中形成紧凑判断；无法可靠解释时先切换 Explore。

没有持久化实现设计不一定不能做。直接改变应当足够简单，能在当前上下文中讲清方案和验证；如果涉及多个功能段、复杂数据或状态变化、显著风险、跨会话推进，先建议建立 issue 并切换 Design，不要边猜边写。

### 实现和验证

按已确认目标、范围和实现判断修改代码。有 issue 时逐项兑现已选质量目标；直接改变时继承相关 spec 约束并处理必要风险。执行发现需要用户决定的阈值、显著成本、兼容策略或目标冲突时停下来说明；复杂度增长到不再适合直接路径时，建议转为 issue，但不未经用户同意自行建档。

UI 变化先确认所属图标记为目标而非纯示意，并只实现本 issue 的局部范围。验证至少覆盖图中构成稳定约束的主路径和本次相关关键状态；截图可以帮助人工对照，但不能替代交互行为、可访问性或状态变化验证。

如果实现采用有界简化，确认它没有破坏目标和质量承诺。有 issue 时把已知上限、升级触发和升级方向补回；直接改变若该上限会持续影响维护者，则写入相关 spec / note 或向用户说明，不为此自动创建 issue。

如果实现设计里包含详细验证策略或 test-first 顺序，按它执行；否则为每个已选质量目标选择与失败代价匹配的证据，再完成 issue 类型要求的必要验证，不临时发明企业级测试计划。

实现后运行最相关的测试/检查。能跑自动验证就不要只手测；不能跑时说明原因和手动验证步骤。

如果下一步需要重新设计、补详细验证策略或用户确认危险操作，停下来汇报当前完成部分和阻塞点。

## 产物契约

受管理路径完成后更新目标 issue：

- `## 执行记录`：改了什么、为什么、和设计是否有偏差；出现有界简化时记录其已知上限、升级触发和升级方向；UI 实现偏离目标图时记录原因并先完成规格消歧。
- `## 验证`：实际跑过的命令或手动步骤、结果，以及它们分别支撑哪些已选质量目标。
- `## 关闭回写`：哪些结论应该在关闭时进入 project spec / epic spec 或 `notes/`。

实现发现现状模型或影响范围不准确时，更新原章节并记录证据；不要在执行记录里另存一份互相冲突的系统解释。

直接路径不创建或补写 issue。保留必要代码、测试和用户可见验证结果；只有改变使已记录的当前真相失效时，才同步相关 project spec。直接改变不自动更新 Vision。

受管理路径不要直接把长期结论写进 spec，除非用户明确要求收尾；通常等切换到 Close 模式关闭事项时再毕业沉淀。

## 收尾汇报

对用户汇报时用收束式汇报：先说这次实现主要解决了什么、采用了什么思路、兑现了哪些质量目标、守住了哪些边界、跑了什么验证；再列关键文件作为证据。不要按文件逐条散报，用户需要先理解结果，再看索引。

## 应用场景

- **用户明确要求直接修改。** 目标简单清楚时直接实现和验证，不补造事项。
- **用户指定一个 issue 要实现。** 读取 issue，按设计修改代码并验证。
- **设计已经完成，可以开写。** 执行 `## 实现设计`，必要时做小幅回写。
- **非 bug issue 要实现。** 按目标和设计推进 feature / chore / 小重构类事项。
- **已诊断完成的 bug issue 要执行剩余实现。** 优先确认 Complain 模式已写清反馈回路、根因定位和验证；缺这些就切换到 Complain，不要在 Do 模式里重新猜根因。

不适用：

- **不处理模糊需求。** 还没说清楚，切换到 Talk。
- **不擅自创建或拆分事项。** 工作需要持续管理时提出建议，由用户决定；明确直接改变不建档。
- **不替代实现设计。** 设计缺失且风险较高，先切换到 Design。
- **不替代高风险设计判断。** 用户或团队要求详细测试设计时，先切换到 Design 补进实现设计。
- **不替代 bug 诊断。** 行为跑偏、报错、性能回退等 bug 全生命周期优先切换到 Complain。
- **不负责最终收尾。** 关闭事项和沉淀 project/epic spec、notes 切换到 Close。
