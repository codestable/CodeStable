# Note：记录可复用知识

## 背景

把“这事以后还会用到”放到未来真正会读取的位置，但不要把 `.cs/` 或 Agent 指令写成垃圾桶。短而稳定、几乎每次启动都影响判断的规则进入项目根已有的 `AGENTS.md` 或 `CLAUDE.md`；需要背景、证据或步骤的知识进入 `.cs/notes/`。

## 原则

先做“六个月测试”：半年后仍会影响判断或操作，而且很难只从代码、git diff、README 重新看出来，才值得写进 `.cs/`。代码已经明说的实现细节、一次性流水账、临时情绪和容易从历史记录重建的东西，不要沉淀成知识。

Agent 指令由框架自动注入，不是知识库。只放稳定、简短、几乎每次相关会话都会影响判断或操作的规则；复杂内容写 notes，再按需在 Agent 指令中放一行阅读指针。

拿不准时默认写 notes，不要污染 Agent 指令。当前项目主线真相进 project spec；一次变更历史留在 issue 或 epic；未知流程切换到 Make Tools 模式由用户带路跑通。

遵循文档组织原则。notes 要服务未来读者的复用任务：标题、结论、触发场景和证据要帮助搜索与判断，不要把一次聊天或一次执行流水账平铺成知识。

## 行动指南

先复用框架已经注入的项目约束和当前上下文，确认相关 notes、project/epic spec 和 issue/epic 是否已经掌握；没掌握、疑似变化、需要查重或准备写回时再读。搜索 `.cs/notes/`、project spec、相关 epic spec 和 issue/epic，避免重复记录；发现已有相近 notes 时，确认当前版本后更新原文件并在末尾加一节日期更新。

写启动规则时优先更新项目已有的 `AGENTS.md` 或 `CLAUDE.md`。跨 Agent 规则优先放 `AGENTS.md`；只对 Claude 生效的规则放 `CLAUDE.md`；两者都存在时不要重复。都不存在时，先向用户说明建议位置，再创建 `AGENTS.md`。写入前先查重，已有相近规则就更新原条目。

旧项目若存在 `.cs/facts.md`，只在用户要求迁移或当前任务确实依赖它时读取。把短规则迁入 Agent 指令，把背景和流程迁入 notes；不要自动删除旧文件，删除前再次确认。

如果信息来自代码、日志或命令结果，先保留可复核证据，不要只写口头结论。

适合写进 Agent 指令的内容包括项目固定测试命令、构建前置、环境陷阱、关键目录禁区、长期命名约定，或“做 X 前先读某篇 note”的索引。保持指令短而可执行，不复制 notes 正文。

写 notes 时，在 `.cs/notes/YYYY/MM/DD/{短语}.md` 写纯 markdown。标题和正文保留用户会搜索的关键词、模块名、命令名、错误名。

## 产物契约

`AGENTS.md` / `CLAUDE.md` 只能追加或更新启动必需的短规则；写入前先确认当前版本并查重，不追加第二条相似说法，不擅自改写无关指令。

`.cs/notes/YYYY/MM/DD/{短语}.md` 推荐结构：

```markdown
# {知识点}

## 结论

{以后最应该复用的判断或做法}

## 触发场景

{这条知识在哪类任务、错误、流程或模块里适用}

## 证据和细节

{代码位置、命令、日志、截图描述、操作步骤、失败分支、危险边界}

## 相关位置

- {issue / epic / spec / 文件路径}
```

一次处理一个主题；多个无关主题拆成多条 notes 或独立的 Agent 指令更新。

## 收尾汇报

告诉用户写到了 `.cs/notes/` 还是 `AGENTS.md` / `CLAUDE.md`、为什么选择这个去处、是新增还是更新已有文件。不要把文件清单当主要汇报，先讲这条知识未来会怎么帮上忙。

## 应用场景

用户说“记下来”“沉淀一下”“留个 note”“以后每次都要知道”，或一次工作暴露出可复用坑点、操作经验、调研结论、长期约束、命令/路径/环境陷阱时使用。

不替代 Close、Make Tools 或 Spec 模式。
