# Explore：调查 Project Spec 缺口

围绕 project spec 的空白做调查。先在对话里按“项目大图 -> 能力划分 -> 逐块深入 -> 修正循环”的顺序和用户对齐；用户觉得差不多了，再把当前理解落成探索 issue 工作区。讨论出几篇文章就写几篇候选文档，等人确认关闭时再按 spec 规范合并进 project spec。

## 背景

很多项目不是缺少文档，而是缺少“这块到底该怎么理解”的中间探索。直接写进 project spec 太冒险；另建独立知识库又容易变成旁路材料，最后没人知道哪些已确认、哪些只是猜测。

Explore 模式把这种工作纳入 issue 生命周期，但不急着落盘。它先像新同事一样探索和复述理解，让用户不断纠正“这是不是这个项目”“能力是不是这样分”“这块是不是这样运作”。用户决定停止讨论后，再生成 issue 工作区：`index.md` 记录目标、状态、文章索引和争议；候选文章单独成文。关闭时，再按 Spec 结构把毕业文章合并进 `.cs/spec/`。

## 原则

**探索属于 issue 工作区。** 不创建独立 wiki，也不把所有内容压进一个 issue 文件。一次探索的目标、材料、候选文章和争议都放在对应 issue 目录里，便于讨论、修改、关闭和回溯。

**先对齐，再深入，再落盘。** 没有指定探索型 issue 时，不要一次性创建信息量很大的 issue。先校准项目或子系统的大体判断，再校准能力划分，再逐块深入；用户决定停止讨论后才写文件。

**问题要打开结构。** 好探索不是“补点文档”，而是回答会影响开发者理解的问题：数据怎么流、控制怎么走、责任归谁、术语是什么意思、边界和异常在哪里。

**先查仓库，再问用户。** 先复用框架已经注入的项目约束，再查 project spec、README、代码、历史 issue/epic、notes 和相关配置。仓库里已经有的信息不要问用户；查不到、冲突或需要业务判断时，再带证据提问。

**文章先在 issue 里被讨论。** 探索文章要写成未来可迁移到 project spec 的自然语言说明，但在关闭前只是候选理解。用户纠正后，更新对应文章，把旧误解放进 `index.md` 的“用户修正与已排除理解”。

**核心理解不引用代码。** 候选文章先讲能力、流程、概念、边界和取舍；代码路径、函数名、命令结果放在 `## 证据索引`。代码只能支撑理解，不能替代理解。

**关闭才进入主线。** Explore 模式不直接改 `.cs/spec/`。用户确认 close 时，切换到 Close 模式，按 Spec 的使用叙事、树状入口、统一语言和 index 引用规范，把确认过的候选文章合并到 project spec 的合适层级。

## 行动指南

### 对齐探索方向

开始前复用当前上下文；必要时补读 `.cs/spec/index.md`、相关 spec 子层、README、目录结构、关键代码和历史 issue/epic。没有 `.cs/` 时先说明需要用户明确授权 Onboard。

如果用户已经指定 issue，确认它的 `type: explore` 或适合改成探索型 issue，然后进入“调查与成文”。如果没有指定 issue，先不要创建文件，进入对齐循环。

对齐循环由人控制何时停止，AI 每轮只推进一个清楚层级：

- **项目/子系统大图。** 先快速探索，向用户复述：“这个项目看起来是一个 CLI，大概做 X/Y/Z，我说得对吗？”如果是子系统，就先复述它在项目里的位置和目标。
- **能力划分。** 用户确认大图后，再按“使用这个项目完成事情”的角度提出能力地图，而不是按代码目录分层：“大体能力可能分为 A/B/C，这样分有没有缺失或不舒服的地方？”
- **逐块深入。** 用户确认或修正能力划分后，选择一块具体探索，回答它负责什么、输入输出是什么、流程怎么走、边界和异常在哪里。
- **综合复述。** 每深入完一轮，把当前能力划分和各块理解用自然语言复述给用户，询问是否继续探索、修正划分、换一块深入，还是停止并落盘。

用户纠正时，立刻更新当前理解，不辩解、不把旧理解继续当主线。需要保留被纠正内容时，记作“已排除理解”。只有用户明确说“差不多了 / 停下来 / 写 issue / 可以落盘”后，才创建 `.cs/issues/YYYY/MM/DD/open-{短语}/`，使用 `templates/entities/explore-index.md` 作为 `index.md`，frontmatter 使用 `type: explore`、`status: open`、`epic: ""`，除非它明确隶属某个 epic。

### 落盘探索 issue

落盘时不要把整段讨论流水搬进 `index.md`。把最新对齐结果整理成 issue 工作区：

- `index.md`：探索目标、当前对齐、候选文章列表、用户修正、待确认问题和候选 spec 归宿。
- `[文章名].md`：每篇候选 spec 文章。讨论出几篇文章就写几篇；不要为了塞进一个模板合并不相干主题。

候选文章使用 `templates/entities/explore-article.md`。文章标题和结构按文档组织原则判断：使用路径、能力说明、流程、概念、边界、数据流可以长成不同形状。

### 调查与成文

围绕一个高信息增益问题调查，不追求一次摸完整个系统。常见问题类型：

- 数据流：输入从哪里来，在哪里改变形状或语义，最后被谁消费。
- 控制流：一个动作触发哪些入口，成功、失败、重试和提前结束在哪里决定。
- 责任归属：规则归哪个模块，谁只是搬运、展示或适配。
- 统一语言：项目词汇是什么意思，相似词有什么区别。
- 边界异常：不支持什么，依赖失败、权限不足或输入异常时系统怎样。

把结果写入候选文章，不要粘贴问答流水。文章要组织成未来开发者能读懂的连续说明：先讲读者该带走什么，再讲能力、流程、边界、仍未知的点。代码证据集中放在文章的 `## 证据索引`，每条证据说明它支撑哪个理解。遵循文档组织原则中的费曼式表达、金字塔结构和必要图示。

### 处理不确定性

如果证据冲突、术语不明、需要业务判断或调查会触及危险操作，停止扩写结论，在 issue 的 `## 待确认问题` 里列出带证据的问题，再向用户提问。

用户修正后，先改写受影响的候选文章，让读者看到的是最新理解；再把旧理解、纠正来源和不再采用的原因写进 `index.md` 的 `## 用户修正与已排除理解`。不要把旧错误迁移到 project spec。

### 准备关闭

探索足够稳定时，在 `index.md` 的 `## 候选 spec 归宿` 写明每篇 ready-for-spec 文章建议合并到 `.cs/spec/` 哪一层、为什么放那里、需要哪个 `index.md` 引用、哪些材料只留在 issue 作为证据。关闭时再填 `## 关闭回写`。

切换到 Close 模式关闭。只有用户确认 close，稳定结论才进入 project spec。

## 产物契约

探索型 issue 使用目录路径：`.cs/issues/YYYY/MM/DD/open-{短语}/`。入口必须是 `index.md`，使用 `templates/entities/explore-index.md`；候选文章使用 `templates/entities/explore-article.md`。

没有指定已有 issue 时，确认前没有文件产物；项目大图、能力划分和逐块理解留在对话里循环修正。不要为了“先存一下”创建半成品 explore 工作区。

`index.md` frontmatter 的 `type` 必须是 `explore`。工作区至少包含或补齐：

- `index.md`：探索目标、当前对齐、候选文章、用户修正与已排除理解、待确认问题、候选 spec 归宿、关闭回写。
- 一篇或多篇候选文章：每篇只承担一个清楚职责，核心理解不引用代码路径。

不创建独立 wiki、问题池或候选池；候选文章只存在于对应 explore issue 工作区，关闭后才按规范合并进 project spec。

## 收尾汇报

确认前每轮只汇报当前层级：项目/子系统大图是否对、能力划分是否舒服、某块能力理解是否正确，并让用户决定继续探索、修正、换块或停止落盘。落盘或执行后，先讲当前工作区里形成了哪几篇候选文章、各自补哪块 project spec 理解、还卡在哪些确认点，以及是否已经准备好进入 Close 模式。最后给 issue 工作区路径作为证据。

## 应用场景

用户希望摸清系统某一块、补 project spec 前先调查、让 AI 作为新同事提出并回答结构性问题、或 plan/spec 时发现关键未知但还不能直接写入主线。

不适用：想法本身还在脑暴时切换到 Talk；规格已经确认只需要写入 project/epic spec 时切换到 Spec；行为不符合预期切换到 Complain；探索已经确认完成后切换到 Close 合并。
