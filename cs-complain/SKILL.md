# cs-complain

`cs-complain` 是 LITE 的 bug 入口：坏了就走这里。它把 main 里的 issue report / analyze / fix 压成一个技能，把 `diagnosing-bugs` 的反馈回路纪律压进执行过程，但只写一个可关闭 issue 文件。

## 总原则

bug 不是从“猜哪行错了”开始，而是从“系统在哪个可观察场景里偏离预期”开始。完整闭环是：

`投诉记录 → 反馈回路 → 复现与最小化 → 假设与定位 → 修复与回归 → issue 回写`

复杂 bug 可以停在任一清楚中间态：已建 issue、已建反馈回路、已最小化、已定位假设、已修复待验证。不要为了显得完成，把没验证的判断写成事实。

## 先确认上下文

1. 如果当前对话/执行上下文已经读过 `cs` 技能，就复用；没读过就先读一次。
2. 读 `.cs/facts.md`；读相关 `requirements/`、已关闭 issue、notes 和代码，先确认“正确行为”是否已经被记录。
3. 如果用户给了截图、日志、堆栈、失败测试、操作路径、请求响应或代码位置，保留原始证据。
4. 搜索 `.cs/issues/` 是否已有同类 bug；有就更新原 issue，不要重复建档。

## 1. 投诉建档

把用户输入拆成四件事：

- **预期行为**：系统应该怎样。
- **实际行为**：系统实际怎样。
- **最小场景**：在哪里、用什么输入或操作，二者开始分叉。
- **证据**：截图、日志、堆栈、请求响应、失败测试、用户反馈、代码位置或最近变更。

只有“已有预期被系统行为打破”才是 bug。预期说不清就继续问；如果是新需求、需求变更或产品取舍不明，转 `cs-talk` 或 `cs-plan`。

投诉成立时，在 `.cs/issues/YYYY/MM/DD/open-{slug}.md` 创建或更新 issue。frontmatter 写 `status: open`，类型写 `bug`，并先填：

- `目标`：修复后可观察的正确行为。
- `范围`：本次 bug 覆盖和不覆盖的边界。
- `当前证据`：预期、实际、最小场景、原始证据来源。
- `验证`：修复后必须回跑的复现步骤、测试或手动检查。

## 2. 建反馈回路

定位前先建立能抓住这个 bug 的反馈回路。优先级：

1. 失败测试：unit / integration / e2e，选能触达真实 bug 模式的最小接口。
2. curl / HTTP 脚本：打运行中的本地服务并断言具体响应。
3. CLI 或脚本：固定输入，diff 输出或状态。
4. 浏览器脚本：Playwright / Puppeteer 驱动 UI，断言 DOM、console、network。
5. 捕获重放：保存真实请求、payload、事件或日志，隔离重放。
6. 临时 harness：只启动最小服务或函数调用路径。
7. 非确定性 bug：循环、并发、加压、固定 seed，把复现率提高到能调试。

反馈回路必须写进 issue 的 `## 反馈回路`：

- 命令或操作入口。
- 它断言的具体症状。
- 最近一次运行结果。
- 是否 red-capable、确定、足够快、agent 可运行。

没有 red-capable 回路时，不进入修复；写清尝试过什么，向用户要日志、HAR、录屏、环境访问或临时埋点许可。

## 3. 复现与最小化

运行反馈回路，确认它抓到的是用户描述的同一个 bug。然后缩小输入、步骤、配置、数据和调用链：一次只删一个因素，删完回跑。留下的每个因素都应该是 load-bearing。

把结论写进 `## 复现与最小化`：最小复现是什么、哪些因素被证明必要、哪些因素被排除。

## 4. 假设与定位

先列 3-5 个可证伪假设，再验证。每个假设都要有预测：

`如果 X 是根因，那么改动/观察 Y 会让症状消失、变重或出现 Z 证据。`

按优先级验证假设。每次只改一个变量。优先用调试器或 REPL；其次用有唯一前缀的临时日志，例如 `[DEBUG-a4f2]`；不要“全量打日志再 grep”。性能问题先建立基线和 profiling / query plan，再谈修复。

把定位写进 `## 根因定位`：

- 假设列表和排序。
- 实际验证过的证据。
- 根因链：触发条件 → 数据/状态/调用怎么变化 → 第一次无效状态在哪里出现 → 为什么导致症状。
- 影响面：哪些模块或用户路径可能同根受影响。

没有证据链时只能写“待定位”，不能写“可能是事实”。

## 5. 修复与回归

修复只消除根因，不补表象，不顺手重构，不引入无关抽象。范围外发现写到 issue 的“顺手发现”，后续另开 issue。

能在正确接缝写回归测试，就先把最小复现转成失败测试，看它失败，再修复。如果没有正确测试接缝，把“缺少可锁定 bug 的接缝”写进 issue；这通常是后续 `cs-design` 或架构改进的信号。

修完必须：

- 回跑原始反馈回路。
- 跑回归测试或手动复现步骤。
- 跑相关影响面的冒烟检查。
- 清理所有 `[DEBUG-...]` 临时埋点和 throwaway harness，除非明确移入 notes/tools。

把代码改动、偏离、验证结果写进 `## 执行记录` 和 `## 验证`。

## 6. 收尾

完成后更新 issue：

- `## 关闭回写`：哪些稳定预期、约束、坑点或流程要进入 requirements / notes / facts / tools。
- `## 关闭结论`：根因摘要、修复摘要、验证摘要、遗留事项。

如果用户要正式关闭，再交给 `cs-close` 做长期实体沉淀；如果只是修到可验证，也可以停在 issue 已更新的状态。

## 边界

- **不主动审计。** 没有具体异常行为，不扫全仓库找问题。
- **不处理新需求。** 预期不存在或要改变预期，转 `cs-talk` / `cs-plan`。
- **不复活三段 issue 技能。** report / analysis / fix 都写回同一个 issue。
- **不要求一次做完。** 复杂 bug 停在清楚中间态，比猜着修完更好。
