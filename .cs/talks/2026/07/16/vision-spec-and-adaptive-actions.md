# Vision Spec 与自适应行动模式 talk

## 原始想法

CodeStable 缺少一个能够记录“我想要这个应用最终变成什么样”的位置。它最初被描述为规划面板，但实际规模可能覆盖整个应用全景，甚至大于单个 Epic。个人开发者希望把尚未成熟的奇思妙想、候选方向和互斥方案放进去，再从中摘取一部分交给 AI 开发。

与此同时，CodeStable 应当教给 AI 一种管理开发过程的方法，而不是要求所有请求强制经过固定流程。用户可能是在直接要求修改、讨论想法、询问现状或规划长期方向，AI 应先理解当前意图，再选择是否使用 CodeStable 的实体与行动模式。

## 真问题

CodeStable 目前能够表达当前稳定真相、局部受控演化和可关闭行动，却没有一个权威位置承载跨越多个 Epic 的目标应用全景；同时，现有叙事容易让 Issue 等实体被理解为开发的必经手续，增加简单改动的管理成本。

需要补齐两项能力：

1. 用一个可递归组织的核心实体保存用户的目标世界，并由 AI 帮助用户把脑内想法整理成可导航的产品地图。
2. 明确 CodeStable 是 AI 的判断与管理参考。AI 先判断用户此刻要提问、讨论、规划还是行动，再按价值选择直接修改、Talk、Vision、Epic、Issue 或其他模式。

## 术语

- Vision Spec：目标中的应用世界。它保存用户希望产品最终具备的体验、能力、边界、候选方案和未决方向，不要求内容已经可以执行。
- Project Spec：当前已经成立的应用世界，只保存当前稳定真相。
- Epic Spec：从目标到现实的一段有边界但仍会演化的变化。
- Issue：确实值得持续追踪、可以验证和关闭的行动；不是每次代码修改的必经步骤。
- Vision 整理：AI 帮助用户定位、分层、关联和重构想法的行动模式，不强迫讨论立即进入开发。
- 自适应行动：AI 根据用户当前意图、事项成熟度、变化规模和持续管理价值选择行动方式。

## 已确认决策

- 核心模型：CodeStable 从三实体扩展为四实体，分别管理目标世界、现实世界、受控演化和可关闭行动。
- Vision 位置：Vision Spec 位于 `.cs/vision/`，由唯一的 `index.md` 建立应用全景和阅读入口，并按需要递归展开。
- Vision 主线：一级组织优先围绕用户旅程，让读者先理解用户如何进入、完成事情并获得结果；能力版图作为辅助视角，与旅程互相链接。
- Vision 内容成熟度：允许目标、候选想法、实验和互斥方案长期并存，不要求所有内容先被收束成确定需求。
- AI 的整理责任：AI 不只是记录原话，还要判断新想法属于现有区域、新产品域还是全局前提变化，帮助用户分成几大块和更小分支，并指出关联、断层与冲突。
- 状态表达：Vision 可以适度标记意图成熟度和实现程度，例如“探索中 / 候选 / 目标 / 放弃”与“未切片 / 已规划 / 建设中 / 部分实现 / 已实现”；具体执行状态仍由 Epic 和 Issue 负责。
- 开发切片：用户准备实现 Vision 的某一部分时，从相关分支摘取内容，经必要澄清后形成 Epic 或 Issue，并保留来源链接。Vision 原文不能被直接当作已明确的实现要求。
- Vision 更新契机：用户可以直接在 Vision 中讨论并确认目标变化；开发过程中不要求每个 Issue 更新 Vision，Epic 经用户确认关闭时再同步实现状态、链接和必要的全景表述。
- 防止静默改写：Epic 关闭时可以按事实更新实现状态和关联；如果实现证据意味着目标方向本身应当改变，AI 必须指出差异并让用户确认，不能静默重写 Vision。
- 意图优先：AI 先判断用户是在要求行动、讨论想法、询问现状还是规划目标，再选择行动模式；不能只凭内容规模机械导向 Vision。
- CodeStable 的约束强度：CodeStable 是供 AI 参考并改变其行动的开发管理方法，不是用户必须完全遵守的强制流程。用户明确要求采用其他方式时，AI 按用户当前请求行动。
- Issue 可选：简单、明确、低管理收益的改变可以直接修改和验证，不必创建 Issue。用户没有明确选择且是否留痕会产生实际差异时，AI 应说明自己的推荐，并询问直接修改还是先建 Issue。
- 明确行动不设障碍：用户已经明确要求直接修改、修复或实现时，AI 不应为了建立 Issue 而阻塞行动；只有复杂度、风险、跨会话推进或项目约束确实需要记录时，才提出相应建议。

## 约束

- Vision 必须与 Project Spec 分离，不能把目标未来写成当前事实。
- Vision 必须能承载比单个 Epic 更大的全景，不能退化为 Epic backlog、任务列表或单页路线图。
- Vision 的全景状态不能复制 Epic 和 Issue 的详细进度，否则会产生多份容易失真的执行真相。
- 文档结构应服务理解路径，不强迫所有旅程、能力、概念和实验套用同一模板。
- AI 可以主动推荐内容归属和行动模式，但用户最新、明确的意图始终优先。
- 是否创建管理实体要考虑其持续价值，不能让记录成本超过变化本身。
- 修改 CodeStable 核心结构或行动模式时，要同步检查相关 references、templates、scripts 和 README，避免同一契约出现两套表述。

## 影响面、风险与取舍

- 核心叙事：`SKILL.md` 的三实体模型、真相关系、辅助实体和行动路由都需要重新表达。Vision 与 Project Spec 不能使用一条简单的优先级覆盖关系，因为二者分别描述目标与现实，可以有意不同。
- Talk 与 Vision：Talk 仍负责把局部想法收束成开发出口；Vision 整理负责扩展和组织目标全景，不要求产生 Epic 或 Issue。AI 需要根据当前意图在两者之间切换。
- Spec 与 Close：Spec 规则需要说明 Vision 的维护边界；Epic Close 需要增加 Vision 状态和链接检查，但 Issue Close 不承担该责任。
- Onboard 与模板：新接入项目需要能够建立 Vision 入口，但不能假设已有项目必须补写完整愿景。Vision 模板应给出地图组织原则，而不是固定 PRD 栏目。
- 向后兼容：已有 `.cs/` 项目可能没有 `vision/`。新版 CodeStable 必须允许这些项目继续工作，并只在用户需要 Vision 时增量补齐。
- 交互成本：如果 AI 对每个简单修改都询问是否创建 Issue，会形成新的仪式负担。只有用户意图不清或记录选择确实影响后续工作时才值得询问；明确的直接行动应直接推进。
- 状态陈旧：Vision 的实现状态更新频率低于 Issue，可能暂时落后于现实。把同步契机绑定到 Epic Close，可以减少维护成本，但 Close 时必须形成明确检查。
- 个人开发者适配：Vision 应允许快速录入和逐步整理，不应要求用户先完整设计产品；AI 的价值在于从碎片中逐渐长出结构。

## 候选质量目标

- 功能适宜性 / 功能完整性：
  - 候选目标：CodeStable 能同时表达目标全景、当前现实、受控演化和可关闭行动，并能从 Vision 的局部形成开发切片。
  - 依据：当前模型缺少跨 Epic 的目标世界，个人开发者的长期构想没有合适归属。
  - 候选证据：用“记录互斥产品想法 → 选取一个区域 → 建立 Epic → Epic 关闭同步 Vision 与 Project Spec”的完整示例验证实体职责。
- 交互能力 / 易操作性：
  - 候选目标：简单明确的修改不会因为 CodeStable 而被迫创建 Issue；AI 能根据用户当前意图推荐并切换行动方式。
  - 依据：强制流程会增加个人开发者的操作成本，并偏离“框架改变 AI 判断”的定位。
  - 候选证据：覆盖直接修改、想法讨论、现状提问、Vision 整理和从 Vision 开发五类对话示例，检查是否得到合适行动。
- 可维护性 / 可分析性：
  - 候选目标：Vision、Project Spec、Epic 和 Issue 各自只有清楚的信息责任，未来读者能判断目标、现实和执行状态应去哪里读取。
  - 依据：新增核心实体容易制造重复规格与状态源。
  - 候选证据：审查核心说明、相关 references、模板、初始化脚本和 README 的契约是否一致，并用冲突场景验证归属规则。
- 兼容性 / 共存性：
  - 候选目标：没有 `.cs/vision/` 的既有项目继续正常使用 CodeStable；Vision 能按需增量引入。
  - 依据：核心结构升级不能要求现有用户立即补写整套产品愿景。
  - 候选证据：在旧结构样例上运行接入或补齐流程，确认不会覆盖既有内容，也不会阻塞普通 Talk、Issue、Epic 和 Spec 行动。
- 灵活性 / 适应性：
  - 候选目标：Vision 文档树能适应不同产品的旅程、能力、概念和探索结构，而无需把所有内容压进统一卡片模板。
  - 依据：目标用户包含以碎片想法逐渐塑造产品的个人开发者，应用全景可能远大于单个 Epic。
  - 候选证据：用两个结构差异明显的产品愿景样例验证同一入口原则能形成不同但清楚的文档树。

## 分歧与待确认边界

- 暂无已确认分歧。
- Vision 根文档及子文档的最小内容契约尚未设计。
- 状态词的最终命名、是否全部进入模板，以及状态在根地图和子文档中的呈现方式尚未决定。
- Vision 整理应作为独立 reference，还是扩展现有 Talk / Spec reference 并由 `SKILL.md` 路由，尚未决定。
- 简单改动场景中，AI 何时必须询问“直接修改还是创建 Issue”、何时可以直接采用推荐路径，还需要在行动规则中写成不过度打扰用户的判断边界。
- 是否未来提供 Markdown 之外的可视化规划面板尚未讨论；当前已确认的是 `.cs/vision/` 文档树与 `index.md` 全景入口。

## 最大未知

如何为 Vision 设计足够明确、可由 AI 稳定维护的文档契约，同时保留个人开发者自由构想和不同产品自然生长的空间，是实现前最大的未知。

## 最终出口

- 出口：直接改变，不创建 Epic 或 Issue。
- 用户决定：Talk 整理后，用户明确要求跳过 Epic，直接修改 CodeStable。
- 判断：虽然变化会改写核心实体模型并涉及多个文件，但目标和边界已经在本轮对话中确认，可以在一个连续执行上下文中完成；额外建档不会增加足够的持续管理价值。
- 实施目标：建立四实体核心模型；定义 `.cs/vision/` 的递归文档契约；新增 Vision 整理行为；让 Talk、Spec、Close、Onboard 和普通开发行动按用户当前意图协作；明确 Issue 是有持续管理价值时才使用的可选实体。
- 验证点：初始化生成 Vision 与 Project Spec 入口、重复初始化不覆盖、技能仓库检查、Skills CLI 发现，以及四实体与直接改变口径的一致性搜索。
