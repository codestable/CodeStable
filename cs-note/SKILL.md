# cs-note

把“这事以后还会用到”落进 `.cs/`，但不要把 `.cs/` 写成垃圾桶。

## 原则

先判断这条信息该去哪里：

| 去处 | 判断标准 |
|---|---|
| `.cs/facts.md` | 一两行能讲清、稳定、几乎每次会话都会影响判断或操作。 |
| `.cs/notes/YYYY/MM/DD/{slug}.md` | 需要背景、证据、步骤、原因或复盘，未来靠全文检索找回来。 |
| 不归本技能 | 当前需求真相进 `requirements/`；一次变更历史留在 issue；未知流程由 `cs-maketools` 带路跑通。 |

拿不准时默认写 notes，不要污染 facts。facts 是启动索引，不是知识库；复杂内容写 notes，再在 facts 加一行指向它。

先做“六个月测试”：半年后仍会影响判断或操作，而且很难只从代码、git diff、README 重新看出来，才值得写进 `.cs/`。代码已经明说的实现细节、一次性流水账、临时情绪和容易从历史记录重建的东西，不要沉淀成知识。

## 开始前

1. 如果当前对话/执行上下文没读过 `cs` 技能，就先读一次，理解 `.cs/` 实体边界。
2. 读取 `.cs/facts.md`；不存在但 `.cs/` 存在时，可以按模板创建最小 `facts.md`。
3. 搜索 `.cs/notes/`、`.cs/requirements/`、相关 issue/epic，避免重复记录；发现已有相近 notes 时，优先更新原文件并在末尾加一节日期更新。
4. 如果信息来自代码、日志或命令结果，先保留可复核证据，不要只写口头结论。

## 写 facts

只写短、稳、启动必读的事实。每条最多两行，优先放到合适小节；没有小节就追加到“启动必读事实”。

适合：

- 项目固定测试命令、构建前置、环境陷阱。
- 关键目录禁区或长期命名约定。
- 已跑通流程的索引：`- 做 X 前，先读 .cs/notes/YYYY/MM/DD/{slug}.md。`

不适合：

- 某个 bug 的修法。
- 多步骤流程全文。
- 还没稳定的临时 workaround。
- 需要解释背景和证据的判断。

写入前先查重。已有相近事实时更新原条目，不要追加第二条相似说法。

## 写 notes

在 `.cs/notes/YYYY/MM/DD/{slug}.md` 写纯 markdown。没有目录就创建。

推荐结构：

```markdown
# {知识点}

## 结论

{以后最应该复用的判断或做法}

## 触发场景

{这条知识在哪类任务、错误、流程或模块里适用}

## 证据和细节

{代码位置、命令、日志、截图描述、操作步骤、失败分支、危险边界}

## 相关位置

- {issue / epic / requirements / 文件路径}
```

notes 可以轻，但要能搜到：标题和正文里保留用户会搜索的关键词、模块名、命令名、错误名。

## 边界

- **不替代 `cs-close`。** issue 完成后的需求和约束毕业，仍由 `cs-close` 分流到 requirements / notes / facts。
- **不替代 `cs-maketools`。** AI 不知道的内部流程，要先由用户带路跑通；本技能可以记录已确认的结论，但不假装会操作。
- **不替代 requirements。** 当前仍有效的需求、规则和取舍，不长期只放 notes。
- **不批量塞信息。** 一次处理一个主题；多个无关主题拆成多条 notes 或多次 facts 更新。

## 结束输出

告诉用户：

- 写到了 `.cs/notes/` 还是 `.cs/facts.md`。
- 为什么选择这个去处。
- 是否更新了已有文件或新增文件。
