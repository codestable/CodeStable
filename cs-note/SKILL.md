## 什么进 cs-note

判据：**长度 + 频次 + 稳定度**，三条都过才进。

| 项 | 进 cs-note | 走别处 |
|---|---|---|
| 长度 | 一两行讲清 | 超半屏 / 需展开 → cs-keep |
| 频次 | 几乎每次会话都用 | 只在某类任务相关 → cs-keep |
| 稳定度 | 项目长期硬约束 | 临时 workaround → 写进 issue |
| 拍板 | 已既成事实 | 选型理由 / 拒方案 → cs-keep；架构决策理由 → cs-context |

✅ 典型该进："编译要先 `pnpm run gen`"、"起服务前 `docker compose up redis`"、"别用 npm install，锁文件是 yarn berry"、"src/legacy/ 改前先问"、"OPENAI_KEY 走 1Password"

❌ 不该进：某 bug 修法 / 某库用法（→ cs-keep）、架构说明（→ cs-context）、"本周在做 X"（短期状态会过期）、3 行以上才讲清的

判不准反问："这条以后每次会话都要让 AI 知道吗？"答"不一定"→ 不是 cs-note。

## 固定分节（写死，不在列表的不开新节）

```markdown
## 项目碎片知识
<!-- cs-note managed -->
### 编译与构建
### 运行与本地起服务
### 测试
### 命令与脚本陷阱
### 路径与目录约定
### 环境变量与凭证
### 其他
```

- 新条目去对应分节末尾追加，每条一行（最多两行）
- 没合适分节 → "其他"；"其他" ≥ 5 条停下来和用户讨论是否新增固定分节（不默默加）
- 分节为空整段保留不删；`<!-- cs-note managed -->` 是识别锚，找不到就在文件末尾插整块
- 整段 ~150 行软上限，超过提示挑几条沉 cs-keep

## 流程

1. **判该不该进**——对照判据表，任一不过引导到别处
2. **找位置**——读 attention.md 定位锚，分节归类；段内查重，有相似条目则更新
3. **写一条**——格式 `- {一句话事实 + 必要时一句话原因}`。写完用户 review 确认。
4. **软上限检查**——≥ 150 行提示沉 cs-keep；"其他" ≥ 5 条提示讨论新增分节。只提示不替决定
